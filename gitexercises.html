While GitHub is a hosting platform for Git repositories, Git is a that can be installed locally to track changes to files, can be used offline, and does not require hosting platforms.

Git is not an organizational convention or folder structure. Git does not rely on manual file naming or back-ups. Rather, Git is a command-line tool or hosting platform that creates or stores a hidden .git folder in your project directory and automatically stores version history, branches, and metadata.

Exercise 1: Create a Practice Folder

In this first exercise, we will create a practice folder on your computer’s hard drive. This is where your file and .git folder will be stored. Follow along, watching both the command line and new folder.

  1. Open Terminal/Command Prompt

    2. Mac: open Terminal from Applications/Utility folder or by using Spotlight Search

    Windows: open Command Prompt by typing "cmd" into Windows search bar

  2. Type 'mkdir git-practice' and hit enter

    3. 'mkdir' is a basic command-line tool, which stands for "make directory." This will create a new folder (repository). On your harddrive, you will see a new git-practice folder.

  3. Type 'cd git-practice' and hit enter

    4. 'cd' is a basic command-line tool, which stands for "change directory". This instructs Terminal/Command Prompt to work inside only this new folder.

  4. Type 'git init' and hit enter to initialize Git

    5. This action will create an invisible .git file within the `git-practice` folder, where versions and metadata will be stored.



Exercise 2: Create a Practice File

Now that we have a folder, we will create a file to go inside. This can be done entirely by command line. We will be creating a markdown file (.md).

! - If you're starting from the last exercise, skip to step 3.

  1. Open Terminal/Command Prompt

    2. Mac: open Terminal from Applications/Utility folder or by using Spotlight Search

    Windows: open Command Prompt by typing "cmd" into Windows search bar

  2. Type 'cd git-practice' and hit enter

    3. 'cd' is a basic command-line tool, which stands for "change directory". This instructs Terminal/Command Prompt to work inside only this new folder.

  3. Type `echo "# This Is A Test" > README.md` and hit enter

    4. `echo` is used to output text, in this case writing the content "This Is A Test" as a title header (#) into a file named README.md.

    You will now see a markdown file named README.md in your git-practice folder, with the content “# This Is A Test”.

  4. Type `git add README.md` and hit enter

    5. `git add` tells Git to prepare for a change you want to save, but does not save the changes yet. `README.md` is the specific file being prepared. There will be no visible change to the markdown file.

  5. Type `git commit -m "Initial commit with README"` and hit enter

    6. `git commit` saves a snapshot of the file and `-m` lets you add a description of the change. In this case, the message is "Initial commit with README".

Exercise 3: Changing a File

Exercise 4: Create a Branch

An important feature of Git is branching. Branches allow us to make edits to existing repositories without overwriting any previous work.

! - If you're starting from the last exercise, skip to step 3.

  1. Open Terminal/Command Prompt

    2. Mac: open Terminal from Applications/Utility folder or by using Spotlight Search

    Windows: open Command Prompt by typing "cmd" into Windows search bar

  2. Type `cd git-practice` and hit enter

    3. 'cd' is a basic command-line tool, which stands for "change directory". This instructs Terminal/Command Prompt to work inside only this new folder.

  3. Type `git checkout -b branch-readme`

    4. `git checkout` is used to switch branches while `-b` tells Git to create a new branch named "branch-readme".

  4. Type `echo "- This is a practice branch" >> README.md` and hit enter

    5. `echo` is used to output text, in this case writing the content "This Is A Test" as a title header (#) into a file named README.md.

    At this point, you will see the new line in your markdown file.

  5. Type `git add README.md` and hit enter

    6. `git add` tells Git to prepare for a change you want to save, but does not save the changes yet. `README.md` is the specific file being prepared. There will be no visible change to the markdown file.

  6. Type `git commit -m "Added note about practice branch"` and hit enter

    7. `git commit` saves a snapshot of the file and `-m` allows you to add a description of the change.

  7. Type `git checkout main` and hit enter

    8. This will switch back to main branch.

    Note: if `main` returns an error, try `master` instead.

  8. Type `git diff main branch-readme`

    9. git diff` compares the differences in branches.

    Note: if `main` returns an error, try `master` instead.

    The new line from your branch will no longer be visible in your markdown file.



Exercise 5: Merging Branches

Once content has been finalized on the alternate branch, it is time to merge the branches together. This allows the main branch to remain updated with the latest version of the content.

! - If you're starting from the last exercise, skip to step 4.

  1. Open Terminal/Command Prompt

    2. Mac: open Terminal from Applications/Utility folder or by using Spotlight Search

    Windows: open Command Prompt by typing "cmd" into Windows search bar

  2. Type `cd git-practice` and hit enter

    3. 'cd' is a basic command-line tool, which stands for "change directory". This instructs Terminal/Command Prompt to work inside only this new folder.

  3. Type `git checkout main` and hit enter

    4. This will switch back to main branch.

    Note: if `main` returns an error, try `master` instead.

  4. Type `git merge branch-readme` and hit enter

    5. This will combine changes from the branch file into the main file.

    If you’re following from the last exercise, your markdown file will now include the line “ - This is a practice branch”.

  5. Type `cat README.md` to see your changes in Git

Exercise 6: Revise in Markdown Editor

Drafting a document entirely in the command line would be an onerous task. That doesn’t mean you have to give up the version control power of Git. In this next exercise we will revise our markdown file in a markdown editor. Once the file has been saved, we can tell Git to track these changes as well.

! - If you’re starting from the last exercise, skip steps 3 and 4

  1. Open your README.md file in a markdown editor

  2. On a new line, type `## Save vs Commit`, then save and close the file

    3. This new line has been saved to your README.md file, but is not being tracked by Git.

  3. Open Terminal/Command Prompt

    4. Mac: open Terminal from Applications/Utility folder or by using Spotlight Search

    Windows: open Command Prompt by typing "cmd" into Windows search bar

  4. Type `cd git-practice` and hit enter

    5. 'cd' is a basic command-line tool, which stands for "change directory". This instructs Terminal/Command Prompt to work inside only this new folder.

  5. Type `git status` and hit enter

    6. This will show that Git sees the change, but the change is not being tracked.

  6. Type `git add README.md` and hit enter

    7. `git add` tells Git to prepare for a change you want to save, but does not save the changes yet. `README.md` is the specific file being prepared. There will be no visible change to the markdown file.

  7. Type `git commit -m "Made changes in markdown editor"` and hit enter

    8. `git commit` saves a snapshot of the file and `-m` allows you to add a description of the change.

Exercise 8: Revising a Commit Message

If you were to mislabel your commit message, perhaps naming it something too vague, there’s still time to change it.

! - If you're starting from the last exercise, skip to step 3.

  1. Open Terminal/Command Prompt

    2. Mac: open Terminal from Applications/Utility folder or by using Spotlight Search

    Windows: open Command Prompt by typing "cmd" into Windows search bar

  2. Type `cd git-practice` and hit enter

    3. 'cd' is a basic command-line tool, which stands for "change directory". This instructs Terminal/Command Prompt to work inside only this new folder.

  3. Type `echo "This line is a mistake" >> README.md` and hit enter

    4. `echo` is used to output text, in this case writing the content "This line is a mistake" into a file named README.md.

    `>>` serves to amend the file, as opposed to `>`, which will overwrite

    Your markdown file will now include the content “This line is a mistake”.

  4. Type `git add README.md` and hit enter

    5. `git add` tells Git to prepare for a change you want to save, but does not save the changes yet. `README.md` is the specific file being prepared. There will be no visible change to the markdown file.

  5. Type `git commit -m "Oops"` and hit enter

    6. `git commit` saves a snapshot of the file and `-m` allows you to add a description of the change.

  6. Type `git log --oneline` and hit enter

    7. This will show the "oops" commit and the HEAD of the list.

  7. Type `git reset --soft HEAD~1` and hit enter

    8. This will reset the commit.

  8. Type `git status` and hit enter

    9. Here you will see that previous changes are still ready to recommit.

  9. Type `git commit -m "Revised commit message"`

    10. `git commit` saves a snapshot of the file and `-m` allows you to add a description of the change. The message on your Git commit has now been changed. The content "This line is a mistake" is still in the markdown file.

  3. Type `echo "This line is a mistake" >> README.md` and hit enter

    `>>` serves to amend the file, as opposed to `>`, which will overwrite.

  4. Type `git add README.md` and hit enter

    5. `git add` tells Git to prepare for a change you want to save, but does not save the changes yet. `README.md` is the specific file being prepared. There will be no visible change to the markdown file.

  8. Type `cat README.md` and hit enter to view your Git versions

Note: after completing any or all of these Git exercises on your computer, you can get rid of the folder, markdown file, and hidden .git file by deleting the git-practice folder.
